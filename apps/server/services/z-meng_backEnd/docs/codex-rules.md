# z-meng 项目的默认协作规则

本文件作为 `z-meng` 项目的默认协作规则文档。

后续在本项目中编写代码、调整结构、补充迁移、补充文档时，默认遵守本文件；如用户在某次任务中明确提出新的临时要求，则以该次要求为准。

## 总原则

目标优先级如下：

1. 保证功能正确
2. 保证代码清晰、可读、可维护
3. 保证模块职责明确，方便未来扩展
4. 保证函数和方法具备复用性与通用性
5. 保证边界条件、空值、异常路径处理完整

默认参考 `mini-drama` 后端中值得借鉴的工程实践，尤其是：

- 清晰的功能分层
- service / repository / handler 职责分离
- provider 抽象
- prompt template 管理
- 调用日志记录
- 数据模型与业务逻辑解耦

## 代码分层规则

默认按“职责单一”原则划分代码层级，不把多种职责混在一个文件里。

推荐分层如下：

- `handler` / 路由处理层
  负责接收请求、参数校验、调用 service、返回响应
- `service` / 业务逻辑层
  负责组织业务流程、编排多个 repository/provider、封装业务规则
- `repository` / 数据访问层
  负责和数据库交互，不承担业务流程编排
- `provider` / 外部能力适配层
  负责对接大模型、第三方平台、存储、消息队列等外部依赖
- `model` / 类型定义层
  负责清晰表达输入、输出、领域对象
- `migration` / 数据迁移层
  负责结构变更、初始化数据、默认模板等数据库演进
- `docs` / 文档层
  负责记录数据库结构、SQL 语法、协作规则、接口约定

禁止出现以下情况：

- handler 里直接堆大量业务逻辑
- repository 里混入复杂业务规则
- provider 里混入数据库写入逻辑
- 单个文件既负责路由，又负责 SQL，又负责 prompt 拼接

## 模块拆分规则

当某个文件开始承担多个职责时，应主动拆分模块。

默认拆分标准：

- 一个文件只围绕一个明确职责
- 一个函数只完成一类动作
- 一个模块只面向一个业务概念

例如：

- AI 调用能力拆成
  - provider 接口
  - 具体厂商实现
  - provider 工厂
  - prompt template service
  - llm call log service
- 剧集相关逻辑拆成
  - create episode
  - update script
  - update stage
  - list episodes

如果新增功能会让现有文件明显变重，默认优先新建模块，而不是继续往旧文件里堆。

## 可读性规则

所有代码默认追求“快速读懂”，而不是“尽量省行数”。

要求如下：

- 命名清晰，优先表达业务含义
- 少写含糊缩写，除非是团队公认缩写
- 函数参数和返回值含义明确
- 复杂逻辑要拆成辅助函数
- 条件分支多时，优先用早返回减少嵌套
- 相同概念在不同模块中保持统一命名

默认不追求炫技式写法，不为了“高级”而牺牲理解成本。

## 可维护性规则

所有改动默认考虑未来继续迭代的成本。

要求如下：

- 避免把值写死在业务代码中
- 可配置项尽量通过配置、模板、常量或数据表管理
- 相同规则不要在多个地方重复实现
- 为未来可能扩展的能力保留合理接口
- 修改现有代码时，优先沿用项目已有结构，不随意引入另一套风格

对于 AI 相关功能，默认优先使用：

- provider 抽象
- prompt template 表
- llm_call_logs 表

而不是把 prompt、模型名、请求逻辑全部写死在 handler 中。

## 复用性与通用性规则

函数和方法默认应尽量具备复用价值。

要求如下：

- 提取通用逻辑，避免复制粘贴
- 对同类能力做统一封装
- 参数设计尽量保持通用，但不要过度抽象
- 通用函数放到合适层级，不乱放

需要避免两种极端：

- 过度复制，导致未来修改多处同步
- 过度抽象，导致代码看不懂、调用复杂

默认优先“够用的抽象”，不要一步设计成过度复杂的框架。

## 判空与边界处理规则

所有输入、查询结果、外部依赖返回值，都默认进行非空判断与边界保护。

重点要求：

- 请求参数要校验
- `id`、分页、枚举值要校验合法性
- 数据库查询结果要判空
- 外部接口响应要判空
- 数组、对象、字符串要防空值
- 对可选字段要提供安全默认值
- 对 cookie / session / auth 信息要严格校验

默认假设：

- 前端可能传错
- 数据库可能查不到
- 第三方接口可能返回异常格式
- 旧数据可能不完整

因此不能依赖“理想输入”。

## 异常处理规则

错误处理要明确、可定位、可追踪。

要求如下：

- 对用户返回清晰可理解的错误信息
- 对服务端记录有定位价值的日志
- 区分参数错误、权限错误、资源不存在、第三方失败、内部错误
- 不吞掉关键异常

对于 AI 调用，默认至少记录：

- 调用来源
- 使用模型
- 输入内容
- 输出内容
- token 使用情况
- finish reason

## 数据库与迁移规则

所有数据库结构变化默认需要：

1. 补 migration
2. 补 autogen SQL
3. 更新表结构文档

对于初始化数据，例如：

- 默认 prompt template
- 默认配置
- 默认枚举映射

优先通过 migration 写入，而不是手工临时插库。

迁移默认要求：

- 尽量可回滚
- 避免重复插入
- 结构变更与初始化数据分清职责
- 重要字段写注释

## AI 功能规则

AI 相关开发默认遵守以下约定：

- 不把敏感 API key 写进业务代码
- 默认通过配置或环境变量注入 key
- prompt 优先通过模板管理，不写死在 handler 中
- 新 prompt 若属于系统默认模板，优先通过 migration 初始化
- AI 返回若要求 JSON，必须在提示词中明确约束格式
- AI 输出进入业务层前，要做基础校验或解析保护

如果后续出现更多 AI 能力，默认延续以下结构：

- provider
- factory
- promptTemplateService
- llmCallLogService

## 文档同步规则

以下改动默认要补文档：

- 新增数据表或字段
- 新增迁移语法知识点
- 新增 AI 调用方式
- 新增默认 prompt template
- 新增重要接口约定

文档要尽量写成：

- 对新人友好
- 有示例
- 能直接对照项目代码理解

## 代码风格偏好

默认风格如下：

- 代码注释少而准，只解释关键意图
- 优先早返回，减少深层嵌套
- 一段逻辑过长时主动拆函数
- 一组常量、类型、辅助函数按阅读顺序组织
- 优先沿用当前项目已有风格

## 执行约定

后续在 `z-meng` 项目中，如果用户没有额外说明，默认按以下方式执行：

- 优先做分层清晰的实现
- 优先做模块化拆分
- 优先补全判空与异常处理
- 优先抽取可复用逻辑
- 优先同步文档与迁移
- 优先参考 `mini-drama` 后端的优秀做法，但不机械照搬过重结构

## 备注
