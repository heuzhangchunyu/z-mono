# 代码规范

本文档用于约束本仓库内 AI Agent 与开发者的前后端代码实现风格。规范以“必要、可执行、贴合当前仓库”为原则，避免空泛要求。

## 通用规范

- 默认使用 TypeScript，必须补充清晰的类型标注，优先补充类型而不是使用 `any`。
- 改动应尽量小而完整，避免无关重构；如果顺手修复相邻问题，需确保行为不变。
- 命名要表达业务语义，避免 `data`、`info`、`handler2` 这类弱语义名称。
- 同一层级的代码放在同一目录职责内，不跨层直接复用内部实现。
- 复杂逻辑优先拆成小函数；单个函数应尽量保持单一职责。
- 注释只解释“为什么”，不解释显而易见的“做了什么”。
- 不保留调试代码、无用 `console`、注释掉的旧实现或未使用的导入。
- 统一优先使用已有工具链和依赖，非必要不新增依赖。

## 前端规范

适用于 `apps/web` 下 React + Vite + TypeScript 代码。

### 目录与职责

- `components/` 放界面组件，聚焦展示与交互，不直接承担接口拼装职责。
- `services/` 放接口调用与业务请求封装，一个业务域一个目录，例如 `auth/`、`episode/`。
- `lib/` 放通用基础能力，例如请求实例、工具函数、基础适配层。
- `types/` 放跨文件复用的领域类型；仅在单文件内部使用的类型可就近声明。
- 样式文件与组件就近放置，命名与组件保持一致，例如 `HomeShell.tsx` 对应 `HomeShell.less`。

### 组件与逻辑

- 组件默认使用函数组件；组件名、导出名使用 PascalCase。
- 前端项目必须做好组件拆分，单个组件同时承担页面编排、复杂交互、数据请求和大段视图渲染时，应主动拆分为更小的容器组件和展示组件。
- 可复用或有明确职责边界的状态逻辑、副作用逻辑、事件逻辑，应拆分为独立 hooks，避免在页面组件中堆叠大量 `useEffect`、`useState` 和业务处理函数。
- 组件应优先保持“展示”和“数据处理”分离：
  - 页面/容器组件负责组织状态、路由、数据请求。
  - 子组件负责渲染和局部交互，避免隐式依赖外部上下文。
- `props`、hooks 入参、返回值、事件处理函数都必须显式声明类型；可选项才使用可选属性。
- 不要在 JSX 中内联大段业务逻辑，复杂判断先提取为具名变量或函数。
- `useMemo`、`useCallback` 只在确有性能或引用稳定性需求时使用，避免为“看起来规范”而滥用。
- 异步请求、订阅、副作用统一放在合适的 effect 或事件处理函数中，组件渲染阶段不得触发副作用。

### 状态与数据流

- 能用局部状态解决的问题，不提前引入全局状态。
- 同一份状态只保留一个可信来源，避免派生状态和源状态长期并存。
- 请求结果进入 UI 前先做最小必要的数据清洗，不把后端响应结构直接散落到多个组件中。
- 对 loading、empty、error 三类状态要有明确处理，避免只处理成功分支。

### 请求与类型

- 所有 HTTP 请求统一走 `src/lib/request/request.ts` 中的请求实例，不在业务代码里直接创建 axios 实例。
- 每个 service 方法都应声明请求参数类型和响应类型。
- 接口路径、参数拼装、响应解构放在 `services/`，组件内只调用语义化方法，例如 `submitAuth()`。
- 不直接把原始错误对象渲染给用户；先转换成可读、可兜底的提示信息。

### 样式规范

- 样式类名使用稳定、可读的组件前缀，避免过短或全局污染命名。
- 优先写组件局部样式，避免依赖跨组件覆盖。
- 不使用 `!important` 解决结构性问题；优先调整选择器层级和样式组织。
- 间距、字号、颜色优先复用已有设计变量或已出现的视觉尺度，避免随意新增魔法值。

## 后端规范

适用于 `apps/server` 下 TypeScript 服务代码。

### 分层与职责

- 路由层负责参数接收、响应输出，不直接承载复杂业务逻辑。
- 业务逻辑优先下沉到 `services` 或领域模块，避免所有逻辑堆在入口文件。
- 与外部系统交互的逻辑单独封装，避免控制器中混杂网络、存储、格式转换细节。
- 若一个文件同时承担启动、路由、业务、配置四类职责，应优先拆分。

### 接口与错误处理

- 请求参数必须校验，至少校验必填项、类型和边界条件；不要默认相信前端输入。
- 成功响应结构保持稳定，字段命名统一，避免同类接口返回风格不一致。
- 预期内错误要转成明确的 HTTP 状态码与业务提示，不把原始异常直接透出给客户端。
- 统一兜底错误处理，避免遗漏 `await` 或让未捕获异常直接中断请求链路。
- 每次新增或修改接口时，必须同步补充或更新对应接口 `md` 文档，至少说明接口路径、请求方法、请求参数、响应字段、错误码或失败场景。

### 类型、配置与日志

- 环境变量读取要集中管理，提供默认值或启动期校验，避免在业务代码中散落读取。
- 类型优先描述领域模型和接口契约，而不是只在使用处临时拼接匿名对象；新增接口时需同步补齐请求体、响应体和领域对象类型。
- 日志要有上下文，至少能定位模块与异常场景；禁止输出密码、token、cookie 等敏感信息。
- 时间、端口、路径、开关等配置项不得硬编码在多处，统一收敛到配置层。

## 变更要求

- 新增功能时，优先沿用现有目录和模式；仅在现有结构明显不适配时再扩展新层级。
- 修改公共方法或接口结构时，要同步检查调用方，避免“局部改通、全局失配”。
- 提交前至少自检：
  - 是否有未使用代码和导入。
  - 是否补齐必要类型。
  - 是否覆盖成功、失败和空状态。
  - 是否引入重复逻辑或跨层耦合。

## 不建议的做法

- 在组件中直接拼接接口 URL、直接写请求细节。
- 在后端入口文件持续堆叠所有业务逻辑。
- 滥用 `any`、类型断言或忽略错误处理。
- 用复制粘贴扩展相似功能而不抽取公共逻辑。
- 为了追求“整洁”一次性做大范围无业务价值重构。
